shiftsim — how it works

A guide to the model behind the sailing-tactics simulator, and what every field on the control panel actually changes. Each field's ? links straight to the matching section here.

The question this answers: in a given wind, when should you tack or gybe? You set up a wind scenario and a fleet of boats with different decision rules, and the simulator races them on the same course and the same wind so you can see — and measure — which choices win, and why.

The big picture

Every half-second, each boat:

  1. reads the wind at its location (direction + speed);
  2. works out its best sailing angle off the wind from its polar, and so its heading on each tack;
  3. asks its strategy whether to tack/gybe — unless a safety rule (layline, etc.) forces the call;
  4. moves at polar speed (reduced if it's mid-maneuver), and records the moment.

All boats see the same wind, so the comparison is fair. Nothing is random unless you choose a puffy wind, and even that is seeded. There is no boat-on-boat contact or right-of-way — this is a tactics sandbox, not a fleet-racing sim. The one way boats do affect each other is bad air (a boat's wind shadow), which is opt-in and off by default.

Directions & angles

  • Bearings are degrees, 0° = North, 90° = East, clockwise — like a compass.
  • Wind direction is where the wind blows FROM (nautical convention). A "Northerly" is 0°. The windward mark is upwind, i.e. in the direction the wind comes from.
  • TWA (true wind angle) is the angle between the boat's heading and the wind. Upwind it's small (~40°), downwind it's large (~150°).
  • A boat's heading is wind direction ± TWA: + on port tack (wind over the left side), − on starboard (wind over the right).

The course & run settings

The course is windward-leeward: start at the bottom, beat up to the windward mark, run back down to the leeward mark. A boat "rounds" a mark when it gets within the mark's radius, or passes closest to it within a capture radius (so it can't sail straight past).

FieldWhat it controls
beat length (m)Distance from the start to the windward mark, in metres. Longer beats give the shifts more room to matter and amplify the differences between strategies.
lapsHow many up-and-down cycles to sail. 1 = one beat + one run. More laps = more chances for tactics (and noise) to accumulate.
max time (s)The simulation cutoff. A boat that hasn't finished by this time is marked DNF. If a sensible boat DNFs, raise this — or it's a sign a strategy is struggling (see safety nets).

Two settings aren't on the panel but exist in a scenario file: dt (the simulation timestep, default 0.5 s — smaller = more accurate, slower) and sample_every (how often a frame is recorded for the replay).

Wind models

The pattern dropdown picks how the wind behaves over time. Every pattern also has a wind speed (kn) — the base true wind speed in knots, which drives boat speed through the polar.

Steady

Constant direction and speed — the control case. dir (°) sets the fixed wind direction. With steady wind there are no shifts, so a good boat simply sails to the layline and tacks once; tactical strategies should not tack here (if they do, that's a bug).

Oscillating

The classic shifty day: direction swings back and forth like a sine wave around a mean. This is where "tack on the headers" is supposed to pay.

FieldMeaning
mean dir (°)The average direction the wind oscillates around.
amplitude (°)The peak swing each way from the mean. 12 means the wind ranges mean ±12°. Bigger amplitude = shiftier = bigger rewards (and penalties) for tactics.
period (s)Seconds for one full cycle (header → lift → header). Short period = frequent shifts = lots of tacking opportunities; long period = you commit to a tack longer.

Persistent shift

The wind trends steadily one way, then holds — e.g. a sea breeze filling and veering right over the afternoon. Here the winning move is usually to sail toward the new wind first and minimise tacks, not to chase every wobble.

FieldMeaning
start dir (°)Wind direction at the start, before the shift.
total shift (°)How far the wind moves in total. Positive = veers right (clockwise), negative = backs left.
over (s)Seconds across which the shift plays out linearly; after that the wind holds at the new direction.

Puffs / velocity

Wind speed (and a little direction) varies as smooth, seeded noise — puffs and lulls. Useful for studying pressure rather than direction.

FieldMeaning
base dir (°)The average wind direction.
gust ±fractionPeak fractional variation in wind speed. 0.3 = gusts/lulls of ±30% around the base speed.
dir wobble (°)How much the direction drifts as the pressure changes (pressure and direction usually move together on the water).
seedThe random seed. The same seed always produces the identical puff sequence, so runs are reproducible and all boats face the same gusts.

Advanced: a scenario file can also use a composite wind — e.g. an oscillation and puffs layered on top of a persistent trend. The web panel exposes the four base patterns; edit a scenarios/*.json for composites.

The boat: polars & speed

A polar is the core of the boat model: it gives boat speed as a function of the true wind angle (TWA) and the wind speed (TWS). Real boats are slowest head-to-wind, fastest on a reach (~90°), and slower again dead downwind. From the polar the simulator derives the two angles that matter:

  • Best upwind angle — the TWA that maximises velocity made good toward the windward mark (speed × cos TWA). Typically ~40–45°.
  • Best downwind angle — the TWA that maximises VMG toward the leeward mark. Typically ~140–160°.

A boat always sails at the optimal angle for its current leg; the strategy only decides which tack to be on.

FieldWhat it controls
boat speed (kn)The boat's approximate top speed in strong breeze. It scales the whole polar — a higher value is a bigger/faster boat. (Internally this is max_speed; there's also a pointing parameter, fixed at 1.1 in the web UI, that sets how high the boat points upwind.)

You can also load a real boat's polar (an ORC/VPP-style CSV of TWA × TWS) in a scenario file instead of the synthetic one.

Sailing the shift — why shifts matter

A boat sailing upwind holds a constant TWA (its optimal angle). So when the wind shifts, the boat's heading rotates with the wind. That single fact is the engine of everything:

  • A lift is a shift that swings your heading toward the mark — you make better progress; stay.
  • A header swings your heading away from the mark — you're losing ground; the other tack is now favoured.

On starboard tack a right-hand shift (wind veering) is a lift and a left-hand shift is a header; on port it's the mirror. "Tack on the headers" means: always be on the lifted tack.

Key subtlety the model gets right: a header is defined relative to the wind, not to the bearing to the mark. A boat that has simply sailed off to one side of the course also finds the other tack points more at the mark — but that's not a header and shouldn't trigger a tack (otherwise a boat zig-zags even in steady wind). So the strategies measure how far the wind has shifted against you relative to its running mean. Getting back to the middle is the layline's job, not the strategy's.

Maneuvers & their cost

Tacking and gybing are slow: you turn through the wind, lose flow over the foils, and have to rebuild speed. That cost is why over-tacking loses, and it's the whole tension in the "when to tack" question. Each boat models it with two fields:

FieldWhat it controls
tack time (s)How long the boat is slowed while recovering from a tack/gybe. During this window its speed ramps from the recovery factor back up to full. Bigger boats / sloppier crews = longer.
recover speed×The speed multiplier at the worst point of the maneuver. 0.45 means the boat drops to 45% speed and recovers from there. Lower = a more expensive, costlier tack.

The maneuver-log panel shows the approximate distance lost to each tack/gybe, computed from these two numbers and the boat's speed at the time. Try dropping recover speed× to 0.25 and watch a busy boat fall down the ladder.

Strategies — when to tack

The strategy is the decision rule you're studying: given the wind and the mark, should this boat tack/gybe now? The start tack sets which tack it begins the first beat on — which, in a persistent shift or when boats sail the same shifts, can decide who's ahead (see the ladder note).

minimize_tacks

Never tacks voluntarily — sails the long board until the layline forces a single tack to the mark. In a persistent shift, combined with the right start tack, this is the "commit to the favoured side" play and usually wins.

tack_on_header

Stay on the lifted tack: tack once the wind has headed you (shifted against your tack, relative to the mean) by more than the threshold.

threshold (°)How big a header you tolerate before flipping, in degrees of wind shift. Small = twitchy (tacks early, near the median wind); large = patient (waits for a bigger header). In steady wind there's no header, so it never tacks. Note that in a clean oscillation the tack frequency is set by the wind's period — the threshold controls how deep into each header you tack, which is what affects your gain.

tack_at_half_header

A refinement: instead of a fixed threshold, tack at a fraction of the oscillation's amplitude — halfway into the header by default. In a clean oscillation this puts the tack partway between the median heading and the maximum header, which is often the sweet spot.

amplitude (°)The oscillation amplitude the boat assumes (set it to match the wind's amplitude). Leave it to auto-estimate in a scenario file by omitting it.
fractionWhere in the header to pull the trigger. 0.5 = halfway; smaller tacks earlier (nearer the median), larger waits deeper into the header. The boat tacks when it's headed by fraction × amplitude degrees.

tack_on_lift

The classic blunder, kept as a control: it tacks when it's lifted (tacking onto the header). It should reliably lose to tack_on_header in oscillating breeze — a sanity check that the model rewards the right behaviour. Its threshold (°) works the same way, just inverted.

fixed_interval

Tacks on a clock every period (s) seconds, ignoring the wind. Mostly for testing and demonstrations.

How a boat decides each step

The strategy is only part of the decision. Each step the simulator runs this pipeline (a maneuver happens at the first rule that fires, after a short debounce so a boat can't flip every step):

  1. Sailing-away safety — if the boat is pointing more than ~107° from the mark it has badly overstood; force a maneuver and enter "recovery" (strategy suppressed until it rounds). Guarantees a boat can never sail off to infinity.
  2. Fetch lock — if the boat is already laying the mark, never tack away from it.
  3. Near the mark — inside a small radius, only laylines act; no tactical tacks right at the mark.
  4. Strategy — the rule you chose.
  5. Layline — if the other tack now fetches the mark, tack so you don't overstand.
Laylines use a smoothed (mean) wind, not the instantaneous direction. A good tactician lays the mark on the average wind, so a momentary shift near the layline doesn't cause an early/late tack and a blown approach. The estimate adapts slowly enough to ignore oscillations but quickly enough to follow a persistent trend.

Thrash-breaker: if a boat maneuvers too many times in a short window it's fighting the safety net (a pathological strategy). It gets flagged "struggling" (shown as struggled in the results) and is sailed on laylines until the next mark — so even a bad rule finishes, it just finishes last.

Start-line situations

Instead of stacking every boat on top of each other at the origin, you can lay a start line across the start and place named boats along it — then run it and compare alternative reactions to the same situation (e.g. hold your lane vs. tack out for clear air). Turn it on by giving the line a length; set it to 0 for the classic single-point start.

FieldWhat it controls
start line (m)Length of the start line, laid square to the wind across the start (committee-boat end to the right looking upwind, pin to the left). 0 = no line; every boat starts at the origin as before.
line posWhere on the line this boat starts, as a fraction from the pin (0) to the committee (1) end. 0.5 is the middle.
rows back (m)How far to leeward (downwind, behind the line) this boat sits, in metres. A boat parked directly downwind of another starts in its bad air.
length (m)The boat's length overall. It sets how far the boat's wind shadow reaches and how big the boat is drawn.
beam (m)The boat's maximum width — its drawn footprint (and the close-in width of its shadow).
Set it up by dragging. On the course you can drag the boats to where they start, and drag the committee boat (RC) and the pin to set the line's length and bias (its angle to the wind — a square line vs. a pin- or boat-favoured one). Releasing a drag re-sails the situation. The numeric line pos / rows back / start line (m) fields stay in sync, so you can nudge by hand or type exact values. Scroll to zoom the map (or use the −/+/fit buttons under it), and drag the background to pan.

A start-line situation is still just a scenario: the line, each boat's placement, and the bad-air settings all live in the JSON, so it reproduces 1:1 and can be shared or filed with a bug report.

Bad air (wind shadow)

A boat disturbs the air to leeward and astern of it: a wedge of "dirty air" blown downwind that robs the boats behind of wind. Switch model bad air on (a start-line feature) and each boat casts a shadow cone pointing straight downwind; a boat inside it sees less wind and so sails slower — which is why you tack to clear your air. The cone is drawn translucent on the course, and a gassed boat is tinted toward red.

FieldMeaning
model bad airTurn the wind-shadow interaction on. Off by default; with it off, boats never affect each other.
reach (lengths)How far downwind the shadow extends, in boat lengths. 8 ≈ the classic "few boat-lengths of dirty air".
cone angle (°)The half-angle of the shadow wedge. Wider = the dirty air spreads to the side more.
max lossThe deepest wind loss, right at a boat's transom — 0.4 = up to 40% of the wind gone. It falls off to zero with distance and toward the cone's edges.
What the model does and doesn't do. It models the velocity deficit only — less wind, less speed — using the true-wind downwind direction, so it stays deterministic. It does not bend the wind direction (the lee-bow effect) or model contact/rules. Combined shadows from several boats stack but are capped, so a boat is never fully becalmed — a movement safety net, like the layline smoothing. The % in bad air result is the share of the beat a boat spent gassed — the "why" behind a start-line loss.

Ladder rungs & the gauge

Ladder rungs are imaginary lines drawn perpendicular to the wind. Every point on a rung is the same distance "up the course" — so whoever is on the highest rung is winning, regardless of which side they're on.

The faint lines on the course are the rungs. Because they're tied to the wind, they rotate as the wind shifts — and that's the whole point: when the wind shifts, two boats that were even suddenly aren't, because the rungs swung. The boat on the lifted side jumps to a higher rung "for free".

The green ladder gauge on the right edge of the course shows each boat's current rung position, top = ahead. Watch the dots reshuffle on every shift.

Why two similar boats can show parallel tracks: if both correctly sail the shifts they converge onto the same tack and then move identically — so any gap from starting on different tacks is preserved as a constant offset (parallel lines). Start them on the same start tack to compare the strategies themselves; the gauge still shows who the early side-selection put ahead.

Reading the results

Column / panelMeaning
finishElapsed seconds to round the last mark. DNF = didn't finish within max time.
t/gNumber of tacks / gybes. Each one cost time (see maneuvers).
Maneuvers panelEvery tack/gybe with the calculation behind it — the reason it fired, the wind, the header/threshold numbers, and the metres lost. Click one to jump there.
Ladder gaugeLive rung position of each boat (see above).

Headless runs (python -m shiftsim run …) also produce a ladder-gain chart — each boat's metres of gain over a reference boat across time — which makes it obvious which shift a lead was won on.

Reproducibility

A scenario plus its seeds fully determines a run. All boats see the same wind, puffs are seeded, and there's no wall-clock randomness — so the same setup always produces the identical result, and any difference between boats is due to their configuration, not luck. That's what makes the A/B comparison trustworthy.

The footer: build version & feedback

The bar at the bottom of the viewer shows the running build as host · branch @ commit · clean/dirty — read live from the server's git checkout. dirty means the deployed tree has uncommitted edits or sits ahead of its upstream; on a clean deploy it reads clean. Quote this when reporting a problem so we know exactly which version you saw.

The Report a bug and Request a feature links open a pre-filled issue on the project's GitHub — you submit it there with your own GitHub account (nothing is sent through the simulator, and no credentials are stored on it). A bug report automatically embeds the current scenario JSON, so we can paste it back in and reproduce your exact run 1:1 — the whole point of the deterministic model. Tidy up the pre-filled description, add what you saw, and submit.

The deploy page

The deploy status page (linked from the viewer footer) is the operations view. It shows what's running on this box, what's on disk (and whether they differ — i.e. a deploy pulled new code but the service hasn't restarted), the tracked branch, and how many commits the box is behind.

The promotion pipeline shows the three deployment tiers — main → stage → live — with the commit gap between each, and a promotion history read from the dated git tags. Promotions themselves run in CI (the promote.yml workflow); the page only shows them.

Two actions are available: Deploy now (fetch the tracked branch, hard-reset the checkout to it, and restart the service) and Restart service. Deploy is restricted to the trusted main / stage / live branches — it can never pull an arbitrary ref.

Security: the deploy page is not behind any login. On the deployed box it's reachable by anyone who can open /sim/, so anyone on the network can trigger a deploy or restart. This is a deliberate choice for a trusted crew network; because deploy is limited to already-reviewed branches, the worst case is redeploying known-good code, not running something new.
